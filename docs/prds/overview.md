# Maestro — 产品概览

> AI Coding 时代的文档中心开发指挥台。
> 最后更新：2026-03-12

---

## 一句话定位

Maestro 是一款 VS Code / Cursor 插件，将开发者的主操作面从"代码编辑器"转移到"文档 / 任务 / 计划面板"，为 AI Coding 工作流提供一等公民级的支持。

## 核心洞察

当前 AI Coding 工具（Cursor、Codebuddy 等）的 UI 仍然是"代码中心"的：

- 90% 的屏幕给了代码编辑器
- 文档只是文件树里的 `.md` 文件，没有结构化视图
- 任务管理靠人脑记忆或外部工具
- AI 上下文注入靠手动复制粘贴
- 迭代进度不直观

但 AI Coding 时代的实际工作流是：

```
人写文档（意图/需求/架构）→ AI 读文档写代码 → 人审代码/抓测试 → 循环
```

开发者的核心工作已经从"写代码"变成了"写文档、管任务、审结果"。Maestro 为这个新工作流提供专属的操作界面。

## 设计原则

### P1: 文档是一等公民（Doc-Centric）

文档不是代码的附属品，而是开发的主要产出物。Maestro 的 UI 以文档为核心组织，而非以文件树为核心。

### P2: 方法论可产品化（Methodology as Product）

Maestro 不是一个通用的项目管理工具，而是将特定的 AI Coding 方法论（me2ai / ai2ai 文档体系、三级决策边界、迭代工作流）内建为产品功能。

### P3: 不侵入、不锁定（Non-invasive）

- 所有数据存储为标准的 Markdown / JSON 文件，不依赖私有格式
- 不在运行时修改用户的 `.cursorrules` 或其他 IDE 配置文件
- 即使卸载插件，文档体系仍然完整可用

### P4: 文档即指令（Docs as Instructions）

Maestro 不试图"控制" AI，而是把文档本身维护好。用户只需在项目初始化时，在 `.cursorrules` / rules 中一次性声明"按文档来"，之后 AI 的行为质量完全取决于文档质量。Maestro 的职责是让文档易写、易维护、易浏览——文档好了，AI 自然好。

具体来说：
- 初始化时，Maestro 可以帮用户生成一段 rules 声明（一次性写入，之后不再修改）
- 运行时，Maestro 只管理 `docs/` 下的文档和任务，不触碰 IDE 配置
- 上下文管理器的作用是帮用户快速组装文档摘要（复制到剪贴板），用于手动粘贴到新 session

### P5: 渐进式采用（Progressive Adoption）

用户可以只用文档导航功能，也可以深度使用任务看板和上下文生成器。不强制全套方法论。

## 目标用户

使用 AI Coding 工具的"导演型开发者"——他们不想逐行写代码，而是想：

- 定义清楚意图和约束
- 把执行交给 AI
- 在高层面审查和验收

## 需求域

### 域 1: 文档导航（Doc Navigator）

将项目中的文档按方法论分层展示，提供结构化的浏览体验。

| 需求 | 优先级 | 描述 |
|------|--------|------|
| DN-1 | P0 | 侧边栏 TreeView，按 me2ai / ai2ai 分组展示文档 |
| DN-2 | P0 | 点击文档在编辑器中打开 |
| DN-3 | P1 | 文档类型图标区分（PRD / Tech / ai2ai / Rules） |
| DN-4 | P2 | 文档内容预览（hover 或面板） |

### 域 2: 任务看板（Task Board）

解析 `docs/ai2ai/tasks/` 下的 task 文件，提供可视化的任务管理。

| 需求 | 优先级 | 描述 |
|------|--------|------|
| TB-1 | P0 | 解析 task markdown 文件，提取任务列表和状态 |
| TB-2 | P0 | Webview 面板展示当前迭代的任务列表 |
| TB-3 | P1 | 在面板中切换任务状态（自动更新 task 文件中的 checkbox） |
| TB-4 | P1 | 点击任务跳转到对应的 task 文件位置 |
| TB-5 | P2 | 迭代切换（查看历史迭代的任务） |
| TB-6 | P2 | 验收检查清单（从 iteration 文件解析可测试项） |

### 域 3: 上下文管理器（Context Manager）

帮助用户维护高质量的文档体系，并在需要时快速组装文档摘要。

| 需求 | 优先级 | 描述 |
|------|--------|------|
| CM-1 | P0 | 一键生成"新 Session 上下文"摘要（基于 session-context 模板），复制到剪贴板 |
| CM-2 | P1 | 可选择包含哪些文档（勾选 PRD、Tech、Status 等） |
| CM-3 | P1 | 自动附加当前活跃 task 的内容 |
| CM-4 | P2 | 上下文模板自定义 |

### 域 3.5: 项目初始化（Project Bootstrap）

帮助用户一键初始化 Maestro 文档体系和 AI rules 声明。

| 需求 | 优先级 | 描述 |
|------|--------|------|
| PB-1 | P0 | 创建 `docs/` 目录结构（prds/, tech/, ai2ai/） |
| PB-2 | P0 | 生成骨架文档（AGENTS.md, status.md, decisions.md 等） |
| PB-3 | P1 | 生成一段 rules 声明（引导 AI 按文档体系工作），用户确认后一次性写入 |

### 域 4: 状态仪表盘（Status Dashboard）

将 `docs/ai2ai/status.md` 和 `decisions.md` 可视化。

| 需求 | 优先级 | 描述 |
|------|--------|------|
| SD-1 | P1 | 模块就绪度可视化（从 status.md 解析） |
| SD-2 | P1 | 迭代历史时间线 |
| SD-3 | P2 | 已知问题 / 技术债列表 |
| SD-4 | P2 | decisions.md 经验库浏览（按类型筛选 D/L/B） |

### 域 5: AI 边界管理（AI Boundary）——远期

| 需求 | 优先级 | 描述 |
|------|--------|------|
| AB-1 | P3 | Always do / Ask first / Never do 可视化展示 |
| AB-2 | P3 | AGENTS.md 结构化编辑器 |
| AB-3 | P3 | Rules 文件管理 |

### 域 6: 验收管线 (Acceptance Pipeline) —— V2重点

将测试与验收阶段融入文档体系，实现“AI 出卷，人类阅卷”的闭环。

| 需求 | 优先级 | 描述 |
|------|--------|------|
| AP-1 | P0 | 自动生成 Checklist：AI 根据需求自动在 `tasks/` 底部生成测试验收清单 |
| AP-2 | P0 | 测试回归继承：自动将 `e2e-cases.md` 中的核心流程和上轮遗留未测项合并到当次 Checklist |
| AP-3 | P0 | 验收工作台 (Acceptance Desk)：专属验收模式 Webview，支持 UI 沉浸式点击打勾 |
| AP-4 | P1 | 双向同步：面板上的 Checkbox 状态精准反写回 task 的 Markdown 文件 |
| AP-5 | P1 | 缺陷反馈环：验收打叉时生成反馈文案并自动注入 Task，打回进行二次修复 |
| AP-6 | P2 | 全局用例沉淀：归档时自动总结提炼具有长期价值的测试作为全局用例 |

## 演进路线

```
MVP（迭代 1-3）
├── 迭代 1: 插件骨架 + 项目初始化命令（创建文档目录 + 生成 rules 声明）
├── 迭代 2: 文档导航 TreeView
└── 迭代 3: 任务看板 Webview

成长期（迭代 4-6）
├── 迭代 4: 上下文生成器
├── 迭代 5: 状态仪表盘
└── 迭代 6: 任务状态双向同步 + 上下文模板自定义

V2 质效突破（迭代 7-8）
├── 迭代 7: 验收工作台与 Checklist 双向绑定
└── 迭代 8: 基于需求的自动化 Checklist 生成与资产库沉淀

成熟期（迭代 9+）
├── AI 边界可视化管理
├── 多项目支持
└── 与更多 AI IDE 兼容
```

## 非目标（Not Goals）

- 不做代码编辑器功能（VS Code 已经做得很好）
- 不做 AI 对话功能（Cursor/Codebuddy 已经做得很好）
- 不做通用项目管理工具（不是 Jira/Linear 的替代品）
- 不自动修改用户的 IDE 配置文件
- 不引入私有文件格式
